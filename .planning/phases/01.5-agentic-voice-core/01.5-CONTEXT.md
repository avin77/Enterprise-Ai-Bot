<decisions>
## Implementation Decisions

### API Provider
- Stay on AWS Bedrock for ALL Claude calls (haiku + sonnet). Do NOT use Anthropic SDK directly.
- Use `bedrock.converse()` API for both OrchestratorAgent (haiku) and ResponseAgent (sonnet).
- Model IDs: `anthropic.claude-3-5-haiku-20241022-v1:0` (orchestrator), `anthropic.claude-sonnet-4-20250514-v1:0` (response).

### Routing Architecture — Hybrid: Soft Intents + Hard Tools
- Orchestrator routes on 2-3 paths only: **retrieval**, **tool**, **fallback**.
- Intent labels (property_tax, utility_services, permits, etc.) are **soft** — logged for analytics/dashboards but do NOT control routing flow.
- Tool routing is **hard** — Orchestrator prompt contains explicit tool list: `property_lookup`, `utility_balance`, `permit_status`. Routes to tool ONLY for these.
- Adding new topics to RAG works automatically (no intent list update needed for retrieval).
- Adding new tools requires updating the tool list in the Orchestrator prompt.

### Fallback Behavior
- Confidence < 0.7 → safe retrieval (skip intent routing, go straight to BM25). No clarification prompts.
- out_of_scope queries → same retrieval path. If BM25 finds nothing, ResponseAgent says "I don't have that info, contact Jackson County at..."
- No hard rejection for any query — always attempt retrieval first.

### Tool + RAG Enrichment
- When Orchestrator routes to a tool, Response Agent gets BOTH the tool result AND top-3 RAG chunks.
- Enables richer responses: "Your balance is $0, next due April 1st. According to the FAQ, late payments incur a 1% penalty."

### Pipeline Architecture — Adapter Swap (Option C)
- `AgentLLMAdapter` implements `LLMAdapter` interface (same `generate(text, system_context)` signature).
- Internally: OrchestratorAgent routes → RetrievalAgent retrieves → ResponseAgent synthesizes.
- `VoicePipeline` stays unchanged — just swap the LLM adapter at construction time.
- Feature flag: `USE_AGENTS=true` in env to enable AgentLLMAdapter (default: false, falls back to RAGLLMAdapter).
- **Rollback:** Flip `USE_AGENTS=false` in ECS task definition for instant rollback to Phase 1 pipeline.

### AgentLLMAdapter Integration
- KnowledgeAdapter passed at construction time in `runtime.py` (same pattern as RAGLLMAdapter).
- Pipeline's existing RAG step still runs but is **ignored** by AgentLLMAdapter (agents do their own retrieval). ~6ms wasted BM25 call — acceptable for zero pipeline changes.
- Guardrails (`build_response_system_prompt()`) wired into ResponseAgent's Bedrock system prompt.

### Phase 2 Integration
- PII scrubber stays at WebSocket layer (main.py) — no changes needed.
- Guardrails re-wired into ResponseAgent system prompt (not RAGLLMAdapter).
- Both paths (agents and non-agents) are self-contained with their own guardrails.

### Trace Events & Observability
- Every turn emits a trace event to CloudWatch Logs (via Python logging, picked up by ECS awslogs driver).
- **Metadata only** in traces — no user query or response text (PII-safe). Text stored in DynamoDB sessions.
- Trace fields: session_id, turn_id, intent, intent_confidence, routing_target, retrieved_doc_ids, llm_prompt_tokens, llm_response_tokens, llm_latency_ms, tool_calls, total_latency_ms, grounded (bool), grounding_signal, fallback_triggered.
- **Grounding detection:** Pattern-based in production (check for "I don't have" phrases → grounded=false). LLM judge in eval runs (Phase 3).

### Dashboard Views (all from trace events)
- Intent volume breakdown (per-topic query distribution)
- Routing path distribution (retrieval vs tool vs fallback over time)
- Per-intent precision/recall (confusion matrix from evals)
- Cost & latency per turn by routing path
- Fallback rate + unanswered/ungrounded rate as first-class metrics

### Mock Boundaries
- **MockMemoryStore:** No-op class with correct interface. `get_history()` returns `[]`. BUT conversation writes in main.py will use **enriched agent format** (include intent, confidence, routing_target alongside existing fields). Phase 2.5 gets richer data from day one.
- **MockToolAgent:** Static canned data. property_lookup → `{balance: 0, next_due: "2026-04-01"}`. Always the same. Predictable for tests and demos. Phase 5 replaces with real API calls.

### Claude's Discretion
- Internal agent class structure and module layout
- Error handling and fallback behavior within agents
- Trace event emission implementation details
- Test structure and fixture design

</decisions>

<specifics>
## Specific Ideas

- RetrievalAgent is a Python class wrapping BM25 (NOT a Claude call) — same interface as future Claude-backed version
- OrchestratorAgent uses JSON system prompt for intent classification (not tool_use)
- ResponseAgent enforces max_tokens=300 for voice-friendly output
- Agent trace events via Python logging (CloudWatch picks up via ECS awslogs driver)
- Fallback: intent_confidence < 0.7 skips Orchestrator routing, goes directly to Retrieval

</specifics>

## Existing Code Insights

### Reusable Assets
- `backend/app/services/knowledge.py` (DynamoKnowledgeAdapter, MockKnowledgeAdapter): BM25 retrieval — RetrievalAgent wraps this
- `backend/app/services/llm.py` (LLMAdapter interface): AgentLLMAdapter implements this

### Established Patterns
- Adapter pattern: ASR/LLM/TTS all use abstract base + mock + real implementations
- Fire-and-forget writes: DynamoDB conversation writes don't block voice turn
- `runtime.py:build_pipeline()` selects mock vs real based on `USE_AWS_MOCKS` env var

### Integration Points
- `runtime.py:build_pipeline()` — add `USE_AGENTS` branch to select AgentLLMAdapter vs RAGLLMAdapter
- `main.py:voice_ws()` — enriched conversation write (add intent, confidence, routing fields)
- `backend/app/agents/` — new directory for all agent modules
