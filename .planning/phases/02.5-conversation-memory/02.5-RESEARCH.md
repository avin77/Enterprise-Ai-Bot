## DynamoDB Session Table — Full Specification

**Table: `voicebot_sessions`** (created in Phase 1.5, data written in Phase 1.5, history READ here for first time)

```
PK: session_id (String)
SK: hashed_timestamp (String)       format: "{hash_prefix}#{ISO_timestamp}"
TTL: expiry_epoch (Number)          epoch = now_unix + 90*24*3600
```

**Hashed prefix implementation:**
```python
import hashlib
from datetime import datetime, timezone

def make_sk(session_id: str) -> str:
    """Hash prefix 0-9 prevents hot partition under burst writes."""
    prefix = int(hashlib.md5(session_id.encode()).hexdigest(), 16) % 10
    timestamp = datetime.now(timezone.utc).strftime("%Y-%m-%dT%H:%M:%S.%f")[:-3] + "Z"
    return f"{prefix}#{timestamp}"
```

**Why timestamp in SK and not turn_number?** ISO timestamp enables natural sort by DynamoDB SK (lexicographic = chronological for ISO 8601). Turn number requires a counter, which introduces race conditions in concurrent writes.

**GSI: NOT needed.** PK = session_id is sufficient to fetch all turns for a session. There is no cross-session query needed in Phase 2.5.

**TTL setup (one-time):**
```python
dynamodb = boto3.client("dynamodb", region_name="ap-south-1")
dynamodb.update_time_to_live(
    TableName="voicebot_sessions",
    TimeToLiveSpecification={"Enabled": True, "AttributeName": "expiry_epoch"}
)
```

## Multi-Turn History Injection

### Fetching Last 5 Turns

```python
async def get_recent_history(self, session_id: str, limit: int = 5) -> list[dict]:
    """Fetch last N turns for a session, in chronological order."""
    response = await self.table.query(
        KeyConditionExpression=Key("session_id").eq(session_id),
        ScanIndexForward=False,   # newest first
        Limit=limit,
        ProjectionExpression="user_input, assistant_response, #ts",
        ExpressionAttributeNames={"#ts": "timestamp"},
    )
    # Reverse to get chronological order (oldest first for Claude messages array)
    items = list(reversed(response["Items"]))
    return items
```

### Converting to Claude Messages Array

```python
def history_to_messages(history_items: list[dict]) -> list[dict]:
    """Convert DynamoDB session items to Claude API messages format."""
    messages = []
    for item in history_items:
        messages.append({"role": "user", "content": item["user_input"]})
        messages.append({"role": "assistant", "content": item["assistant_response"]})
    return messages

# Usage in Orchestrator:
history = await memory.get_recent_history(session_id, limit=5)
messages = history_to_messages(history)
# messages is now [{role:user,...}, {role:assistant,...}, ...] × 5 turns = 10 messages
# Current turn appended: messages + [{"role": "user", "content": current_query}]
```

### Token Budget Management

5 turns × 2 messages × ~150 tokens avg = ~1,500 tokens of history context.

Claude haiku context window: 200K tokens. Budget is fine.
Claude haiku price: $0.25/M input tokens. 1,500 extra tokens = $0.000375 per turn. Acceptable.

**Truncation strategy:** If a single message exceeds 500 tokens, truncate with `[...truncated for context...]`. Apply to assistant responses (user voice input is always short).

### Session Boundary Detection

```python
SESSION_TIMEOUT_MINUTES = 30

async def get_or_create_session(self, client_id: str) -> str:
    """Return existing active session_id or create new one."""
    # Check Redis for active session (fast lookup)
    cached = await redis.get(f"session:{client_id}")
    if cached:
        # Refresh TTL
        await redis.expire(f"session:{client_id}", SESSION_TIMEOUT_MINUTES * 60)
        return cached.decode()

    # Create new session
    session_id = str(uuid.uuid4())
    await redis.setex(f"session:{client_id}", SESSION_TIMEOUT_MINUTES * 60, session_id)
    return session_id
```

`client_id` = browser fingerprint or WebSocket connection ID for anonymous users. Not a user account ID — privacy-preserving session tracking.

## Conversation Metrics — Derivation Rules

All 6 metrics derived per turn, written to DynamoDB session record:

| Metric | Derivation | Implementation |
|--------|-----------|----------------|
| `follow_up_detected` | Current intent == previous intent AND prior turn's `completion_status != "complete"` | Compare intent from last turn in history |
| `fallback_triggered` | `routing_target == "fallback"` in current turn | Already in agent trace |
| `clarification_requested` | User input contains clarification signals | Regex: `r"(what do you mean|can you explain|i don't understand|clarify|what\?)"` |
| `completion_status` | Heuristic: "thank you" in user input → "complete"; 10+ turns with same intent → "abandoned"; else "in_progress" | Applied at turn write time |
| `turn_number` | Count of items in DynamoDB session (query Count before write) | Or maintain Redis counter |
| `session_duration_ms` | `current_timestamp - first_turn_timestamp` (from earliest session record) | Fetch first turn on session end |

**Where to compute:** Synchronously BEFORE writing session record. These are cheap string operations (~0ms).

## Cost Per Conversation — Calculation

### Per-Turn Cost Formula

```python
# Fargate pricing (ap-south-1 as of 2026):
FARGATE_VCPU_PER_HOUR = 0.04048   # USD
FARGATE_GB_PER_HOUR = 0.004445    # USD
TASK_VCPU = 0.5                    # 512 CPU units = 0.5 vCPU
TASK_GB = 1.0                      # 1024 MB = 1 GB

# Claude pricing (verify against Anthropic pricing page):
HAIKU_INPUT_PER_M = 0.25           # USD per 1M input tokens
HAIKU_OUTPUT_PER_M = 1.25          # USD per 1M output tokens
SONNET_INPUT_PER_M = 3.00          # USD per 1M input tokens
SONNET_OUTPUT_PER_M = 15.00        # USD per 1M output tokens

def calculate_turn_cost(
    turn_duration_ms: int,
    orchestrator_prompt_tokens: int,
    orchestrator_completion_tokens: int,
    response_prompt_tokens: int,
    response_completion_tokens: int,
    dynamo_wcus: int = 2,           # 2 WCU per turn (session write)
    dynamo_rcus: int = 5,           # 5 RCU per turn (history fetch)
) -> dict:
    # Compute cost (proportional to task uptime)
    turns_per_hour = 3600 / (turn_duration_ms / 1000)  # turns if 100% busy
    compute_cost = (FARGATE_VCPU_PER_HOUR * TASK_VCPU + FARGATE_GB_PER_HOUR * TASK_GB) / turns_per_hour

    llm_cost = (
        (orchestrator_prompt_tokens / 1_000_000) * HAIKU_INPUT_PER_M +
        (orchestrator_completion_tokens / 1_000_000) * HAIKU_OUTPUT_PER_M +
        (response_prompt_tokens / 1_000_000) * SONNET_INPUT_PER_M +
        (response_completion_tokens / 1_000_000) * SONNET_OUTPUT_PER_M
    )

    # DynamoDB: $1.25/M WCU, $0.25/M RCU (on-demand)
    storage_cost = (dynamo_wcus / 1_000_000) * 1.25 + (dynamo_rcus / 1_000_000) * 0.25

    return {
        "compute_cost_usd": round(compute_cost, 6),
        "llm_cost_usd": round(llm_cost, 6),
        "storage_cost_usd": round(storage_cost, 6),
        "total_cost_usd": round(compute_cost + llm_cost + storage_cost, 6),
    }
```

**Expected per-turn cost:**
- Compute: ~$0.005 (1 turn taking 2s on a task running 30 turns/hour)
- LLM: ~$0.010 (haiku: 500 tokens + sonnet: 400 tokens)
- Storage: ~$0.000003 (negligible)
- **Total: ~$0.015/turn** (well under $0.03 target)

## Intent Confusion Matrix — CloudWatch Aggregation

**Production (no ground truth):** Compute weekly using CloudWatch Logs Insights over agent traces:

```sql
fields intent, follow_up_detected
| filter event_type = "agent_trace"
| stats
    count() as total,
    sum(follow_up_detected == 1) as follow_ups
    by intent
| eval precision_proxy = 1 - (follow_ups / total)
| sort precision_proxy asc
```

**Eval runs (with ground truth):** Compare predicted_intent vs gold_intent from fixture file → compute true precision/recall.

**Publish to CloudWatch:**
```python
# After each eval run, publish per-intent precision
```

## Don't Hand-Roll

- **Session ID from UUID4:** Use `uuid.uuid4()` — not timestamp-based IDs (collision risk under burst)
- **History fetch order:** Always use `ScanIndexForward=False` + `Limit=N` + reverse in Python — DynamoDB sorts by SK, not by turn_number
- **Decimal vs float:** DynamoDB boto3 requires `Decimal` for numeric types. Use `from decimal import Decimal` throughout cost calculations
- **Redis for session boundary detection:** Don't query DynamoDB to check "is session active" — that's 1 RCU per check × every request = expensive

---

## Common Pitfalls

1. **ISO 8601 sort order in DynamoDB SK:** Must use full precision `"2026-03-10T14:23:01.123Z"` — shorter formats don't sort correctly
2. **History includes current turn:** Do NOT include the current turn in history passed to Orchestrator — only prior turns
3. **Consecutive same-role messages:** If prior assistant response was empty (error), Claude API rejects the history. Filter out empty turns before passing to Claude.
4. **Cost calculation turns_per_hour assumption:** Use actual measured turns/hour from CloudWatch metric — not theoretical max
5. **Session Redis expiry vs DynamoDB TTL:** These are independent. Redis expiry = session activity window (30 min). DynamoDB TTL = data retention (90 days).
