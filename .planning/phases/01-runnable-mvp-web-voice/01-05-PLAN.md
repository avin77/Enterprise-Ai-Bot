# Plan 01-05: Local Observability Dashboard

**Objective:** Create a real-time web dashboard (localhost:8001) that visualizes voice bot behavior locally, enabling PM validation without AWS credentials.

**Why This Matters:** Phase 1 infrastructure (ECS, DynamoDB, CloudWatch) is live but invisible locally in mock mode. PM needs observability to:
- Verify RAG is retrieving correct documents
- See LLM input/output for each turn
- Monitor Redis cache behavior (hits vs misses)
- Trace WebSocket request→response flow
- Validate conversation session tracking
- Check per-stage pipeline latency

---

## Tasks

### Task 1: Create Dashboard HTML/CSS/JS Skeleton
**Objective:** Build single-page dashboard with layout for all metrics.

**What to build:**
- `frontend/pages/dashboard.html` — responsive HTML layout
  - Header: "Voice Bot Dashboard (Local)"
  - 6 sections: RAG Trace | LLM I/O | Redis Status | WebSocket Log | Conversation | Pipeline Metrics
  - Auto-refresh toggle (every 2 seconds)
  - Clear history button
  - Session filter (show last N interactions)
  - Styling: clean, minimal (no framework required — vanilla CSS)

**Acceptance criteria:**
- [ ] Dashboard loads at `http://localhost:8000/dashboard` (or separate port 8001)
- [ ] 6 sections render with placeholder content
- [ ] Responsive layout (mobile/tablet/desktop)
- [ ] No external JS libraries required (vanilla JS only)
- [ ] Styling uses CSS Grid or Flexbox

### Task 2: Connect to /metrics Endpoint & Display Pipeline Latency
**Objective:** Poll the existing `/metrics` endpoint and display real-time latency stats.

**What to build:**
- `frontend/js/dashboard.js` — fetch `/metrics` every 2 seconds
  - Parse JSON response (asr, rag, llm, tts, total)
  - Display in "Pipeline Metrics" section:
    - P50/P95/P99 for each stage
    - Color coding: green (<100ms), yellow (100-300ms), red (>300ms)
    - Last updated timestamp

**Acceptance criteria:**
- [ ] Fetches `/metrics` endpoint successfully
- [ ] Displays p50, p95, p99 for ASR/RAG/LLM/TTS
- [ ] Auto-updates every 2 seconds
- [ ] Shows last 100 entries (rolling window)
- [ ] Error handling if endpoint unavailable

---

### Task 3: Add WebSocket Trace Logger
**Objective:** Capture and display all WebSocket messages (input/output) in real-time.

**What to build:**
- Modify `backend/app/main.py` WebSocket handler to emit trace events
  - New endpoint: `GET /api/traces` — returns JSON array of recent traces
  - Trace format: `{timestamp, session_id, direction, message_type, content, status}`
  - Store last 100 traces in memory (deque)

- Update `frontend/js/dashboard.js`:
  - Fetch `/api/traces` every 2 seconds
  - Display in "WebSocket Log" section as scrollable table:
    - Timestamp | Session | Direction (IN/OUT) | Type | Status | Content preview

**Acceptance criteria:**
- [ ] `/api/traces` endpoint returns recent WebSocket events
- [ ] Dashboard displays last 20 traces in scrollable table
- [ ] Shows IN (client→server) and OUT (server→client) messages
- [ ] Color-coded status (✓ success, ✗ error)
- [ ] Truncates long messages with "..." (expand on click)

---

### Task 4: Add RAG Retrieval Tracer
**Objective:** Display what documents RAG retrieved for each query.

**What to build:**
- Modify `backend/app/orchestrator/pipeline.py`:
  - Log RAG retrieval results: `{query, documents_retrieved, scores, cache_hit}`
  - New endpoint: `GET /api/rag-traces` — returns last 50 retrievals

- Update `frontend/js/dashboard.js`:
  - Fetch `/api/rag-traces` every 2 seconds
  - Display in "RAG Trace" section:
    - Query input
    - Top 3 documents (title, score)
    - Retrieval time (ms)
    - Cache hit/miss indicator
    - Table with sortable columns (score, time)

**Acceptance criteria:**
- [ ] `/api/rag-traces` endpoint returns recent RAG queries
- [ ] Dashboard shows query input and top 3 documents retrieved
- [ ] Displays BM25 relevance scores (0.0-1.0)
- [ ] Shows whether cache was hit or missed
- [ ] Color code: green (cache hit), blue (BM25 search)

### Task 5: Add LLM I/O Tracer
**Objective:** Display system prompt, user message, and Claude response for each turn.

**What to build:**
- Modify `backend/app/services/llm.py`:
  - Log LLM calls: `{user_message, system_prompt, response, latency_ms}`
  - New endpoint: `GET /api/llm-traces` — returns last 50 LLM calls

- Update `frontend/js/dashboard.js`:
  - Fetch `/api/llm-traces` every 2 seconds
  - Display in "LLM I/O" section:
    - User message (with truncate/expand)
    - System prompt (collapsible, show RAG context)
    - Claude response (collapsible, full text on expand)
    - Latency badge (ms)
    - Token count estimate

**Acceptance criteria:**
- [ ] `/api/llm-traces` endpoint returns recent LLM interactions
- [ ] Dashboard shows user input → system context → LLM output
- [ ] System prompt includes RAG documents used
- [ ] Latency displayed (e.g., "245ms")
- [ ] Expandable sections for long text

### Task 6: Add Redis Cache Status Monitor
**Objective:** Display cache hits/misses and fallback events.

**What to build:**
- Modify `backend/app/services/redis_cache.py`:
  - Log cache events: `{event_type, key_hash, hit_or_miss, latency_ms}`
  - New endpoint: `GET /api/cache-stats` — returns stats + last 20 events

- Update `frontend/js/dashboard.js`:
  - Fetch `/api/cache-stats` every 2 seconds
  - Display in "Redis Status" section:
    - Hit rate (e.g., "80% hits, 20% misses")
    - Pie chart (hits vs misses)
    - Event log: timestamp | operation | result | latency
    - Highlight fallback events (when Redis unavailable)

**Acceptance criteria:**
- [ ] `/api/cache-stats` endpoint returns hit/miss counts
- [ ] Dashboard shows hit rate percentage
- [ ] Displays last 10 cache events in timeline
- [ ] Highlights fallback events in red/orange
- [ ] Shows cache latency for successful hits

### Task 7: Add Conversation Session Tracker
**Objective:** Display active conversation sessions and SLO compliance.

**What to build:**
- Modify `backend/app/services/conversation.py`:
  - New endpoint: `GET /api/conversations` — returns all active sessions
  - Response: `{session_id, turn_count, slo_met_count, slo_violated_count, created_at, ttl_remaining, cost_so_far}`

- Update `frontend/js/dashboard.js`:
  - Fetch `/api/conversations` every 2 seconds
  - Display in "Conversation" section:
    - Active sessions table
    - Session ID | Turn # | SLO Met % | TTL Remaining | Cost
    - Click session row to show all turns in that session
    - SLO threshold reminder (1500ms)

**Acceptance criteria:**
- [ ] `/api/conversations` endpoint returns active sessions
- [ ] Dashboard shows session ID, turn count, SLO compliance
- [ ] Calculates SLO met %: `(slo_met_count / turn_count) * 100`
- [ ] Shows TTL countdown (days remaining)
- [ ] Shows cost per session (cumulative)
- [ ] Color code: green (100% SLO), yellow (80-99%), red (<80%)
