# Phase 02: Public Sector Safety — Design Document

**Date:** 2026-03-12
**Status:** Approved
**Phase:** 02 of 10

---

## Summary

Phase 02 adds a full safety pipeline to the voice bot. Every conversation turn passes through PII scrubbing (input + output), content guardrails injected into the Response Agent, and an independent Supervisor Agent that vetoes unsafe responses before they reach the user. All vetoes are written to a tamper-evident audit trail. The phase is gated on 200/200 adversarial red team queries passing automated safety evaluation.

---

## Problem

Phase 01 delivered a working RAG voice bot with zero safety controls:
- User PII (SSN, phone, address) is stored verbatim in DynamoDB and CloudWatch logs
- The Response Agent will attempt to answer legal, medical, and financial questions
- There is no audit trail of safety incidents
- The API has no rate limiting

None of this is acceptable for a government-facing product.

---

## Approach

**Option chosen: Safety module with explicit call chain** (`backend/app/safety/` package)

Each safety concern is a separate, independently testable module. The orchestrator pipeline calls them explicitly at defined points. This was chosen over inline middleware (creates god function) and FastAPI middleware (WebSocket hooks are awkward and Supervisor needs LLM output content).

---

## Architecture

### Safety Pipeline Per Turn

```
User speaks
    ↓
ASR (speech → text)
    ↓
pii.scrub_input(text)               ← strips SSN, phone, address etc.
    ↓
Orchestrator → Retrieval → Response Agent (with guardrail rules in system prompt)
    ↓
supervisor.evaluate(query, draft)   ← Haiku APPROVE or VETO
    ↓
    ├─ VETO  → audit.write_veto()   ← HMAC-signed DynamoDB record
    │          send canned response
    │
    └─ APPROVE → pii.scrub_output(draft)
                 send scrubbed response
    ↓
TTS → user hears response
```

### New File Structure

```
backend/app/safety/
    __init__.py
    pii.py              # Presidio scrubber — input + output passes
    guardrails.py       # Prohibited content rules + prompt builder
```

## Components

### 1. PII Scrubber (02-01)
- Library: Microsoft Presidio + spaCy `en_core_web_sm`
- Entities: PERSON, EMAIL_ADDRESS, PHONE_NUMBER, US_SSN, US_DRIVER_LICENSE, US_PASSPORT, US_BANK_NUMBER, CREDIT_CARD, LOCATION
- Two passes: input (before orchestrator), output (after Supervisor APPROVE)
- Models pre-loaded at container startup — not per-request
- Latency: ~15ms per call
- PRD: `docs/plans/prd/02-01-pii-scrubber.md`

### 2. Content Guardrails (02-02)
- 5 prohibited categories: legal_advice, medical_advice, financial_advice, county_promise, third_party_pii
- Injected as string into Response Agent system prompt on every call
- Zero added latency
- Rules defined once in `guardrails.py` — no duplication
- PRD: `docs/plans/prd/02-02-content-guardrails.md`

### 3. Supervisor Agent + Audit Trail + API Hardening (02-03)
- Model: `claude-haiku-4-5-20251001`, temperature=0
- Evaluates every Response Agent draft — APPROVE or VETO + rule_category
- 7 veto categories with canned safe responses per category
- Audit: HMAC-SHA256 signed DynamoDB records (VETO only), TTL 365 days
- Raw text never stored — SHA-256 hashes only
- Rate limiting: 60/min HTTP, 30/min WS
- Latency cost: ~300ms per turn
- PRD: `docs/plans/prd/02-03-supervisor-agent.md`

### 4. Adversarial Red Team Eval (02-04)
- 200 queries: 10 categories × 20 queries
- Generated by Claude Sonnet (offline), human-reviewed, committed to git
- Judge: Claude Haiku evaluates each response — SAFE or UNSAFE
- Exit code 1 if any UNSAFE — CI-blockable
- **Phase gate: 200/200 required before Phase 03 begins**
- PRD: `docs/plans/prd/02-04-adversarial-eval.md`

## Phase Gate

Phase 02 is complete when:
1. All unit tests pass (`pytest tests/unit/safety/ tests/unit/agents/`)
2. `python evals/phase2_eval.py` exits 0 (200/200 SAFE)

Phase 03 (Eval Gate I) may not begin until both conditions are met.
